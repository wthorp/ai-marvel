# Marvel Fact Recall Experiment Plan

## Working Thesis

Fine-tuning a LoRA is the wrong primary tool for exact factual recall over a 1.9 GB wiki dump. It can learn common patterns and frequent facts, but a retrieval system or typed knowledge graph should win on exactness, provenance, updateability, and debuggability.

The blog should test that claim locally, with timed stages and reproducible artifacts.

## Systems To Compare

1. Plain vector RAG
   - Preprocess MediaWiki XML into pages and chunks.
   - Embed chunks locally, initially with `BAAI/bge-m3`.
   - Retrieve top-k chunks and answer through a local OpenAI-compatible model endpoint.

2. Hyper-Extract Marvel graph
   - Use the same filtered page set as RAG and LoRA.
   - Feed cleaned page records into Hyper-Extract with a Marvel-specific graph template.
   - Compare factual answers from graph search or graph-derived lookup with citations back to source pages.

3. Closed-book LoRA or QLoRA
   - Generate deterministic QA pairs from parsed Marvel templates.
   - Train an adapter locally on an RTX 3090.
   - Evaluate without retrieval against held-out deterministic questions.

## Data Scope

Use staged scaling so failures are informative:

1. Smoke: first 50 matching pages.
2. Small: 1,000 matching pages.
3. Medium: 10,000 matching pages.
4. Large: 100,000 matching pages.
5. Full: all matching pages if earlier stages justify the time.

Matching pages start with namespace 0 and one of these template families:

- `Marvel Database: Character Template`
- `Marvel Database: Comic Template`
- `Marvel Database: Volume Template` later, after character/comic extraction is stable.

## Evaluation

Questions are generated by code, not by a local model.

Question categories:

- Single attribute: current alias, reality, identity, citizenship, gender, release date, story title.
- Single relation: first appearance, creator, writer, event, parent, spouse.
- Multi-answer relation: affiliations, featured characters/entities in an issue.

Scoring:

- Exact normalized match.
- Set exact match for multi-answer questions.
- Partial overlap for diagnostic reporting only.
- Answerable rate.
- Hallucinated answer rate for deliberately unanswerable questions, added after the baseline works.
- Latency per question.
- Provenance availability for RAG and graph systems.

## Timing And Resource Measurements

Every stage should be run through `scripts/run_timed.py`, which records:

- Wall time.
- User and system CPU time.
- Maximum resident memory reported by the OS.
- Command exit status.
- Optional GPU samples from `nvidia-smi` when available.

Timed stages:

1. XML scan and page extraction.
2. Wikitext template parsing.
3. Graph fact projection.
4. Deterministic QA generation.
5. RAG embedding and index build.
6. Hyper-Extract parse.
7. LoRA dataset formatting.
8. LoRA training.
9. Inference and evaluation.

## First Implementation Milestone

Build the common substrate:

1. Extract filtered pages from the XML dump into JSONL.
2. Parse Marvel templates into fields.
3. Project fields into attributes, relations, and deterministic QA.
4. Prepare Hyper-Extract text shards and a Marvel-specific template.
5. Prepare LoRA SFT train/eval splits from the same QA file.
6. Add vector index scaffolding for the RAG baseline.
